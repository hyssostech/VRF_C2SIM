## 4. TRAPS THAT COST TIME TODAY - DO NOT REDISCOVER THEM

- **`Start-Process -Wait` waits for the process AND ALL DESCENDANTS.** It can therefore
  NEVER return from a stage that launches VR-Forces. Cost a 47-minute deadlock. Use
  `-PassThru` + `$p.WaitForExit(ms)`. `Wait-Process` waits only for the named process.
- **A plain MSBuild of VrfBridge returns EXIT=0 in ~2 seconds having compiled NOTHING.**
  Always `/t:Rebuild`. This is a false green of the same shape as the retracted
  `reflected>0` criterion.
- **`grep -P` SILENTLY DEGRADES on this machine's locale** and reports a false clean for
  the ASCII check. Verify in PowerShell with a byte scan. Worse: `grep -P ... && echo CLEAN
  || echo CLEAN` fires the fallback on the ERROR, not on success.
- **`Parser::ParseFile(path, [ref]tokens, [ref]errors)`** - swapping the last two reports
  thousands of "parse errors" that are actually tokens.
- **`[IO.File]` uses the PROCESS working directory, not PowerShell's `Set-Location`.** It
  resolved a path into the FROZEN C++ ORACLE repo once. Always pass absolute paths.
- **`$PID` is read-only** in PowerShell; assigning to it errors per-iteration.
- A `Where-Object { $_.CommandLine -like '*needle*' }` process query MATCHES ITS OWN
  PROCESS, because the needle is in the querying command line. Build the needle at runtime.

## 6. FINDINGS THAT SURVIVED THE REVERT - keep these, they are implementation-independent

- **RUNBOOK sec 7's "RTTI across the MAK DLL boundary" story is FALSE.** MAK's own header

  `DtReflectedObject` instead of `DtReflectedAggregate`, so a null
  `dynamic_cast<DtReflectedAggregate*>` is the cast behaving CORRECTLY. No RTTI mechanism
  is involved. Corrected in RUNBOOK.
